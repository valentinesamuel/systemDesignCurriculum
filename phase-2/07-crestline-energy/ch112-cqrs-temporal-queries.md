**Slack DM — Dr. Marcus Thompson → [you] — Tuesday 08:30**

**Dr. Marcus Thompson**: I've been thinking about the temporal query problem overnight. The core issue is that we never designed for "point in time" queries. The Flink streaming pipeline overwrites state — it doesn't append. So reconstructing "what was the state at T?" requires replaying the event log from scratch.

**[you]**: That's the fundamental issue with mutable state. Every time we write a new meter reading, we overwrite the previous one. To know what it was at 14:32:07, we'd need every reading from 14:17:07 to 14:32:07 and reconstruct from there.

**Dr. Marcus Thompson**: Which we can do with the S3 archive. It's just slow.

**[you]**: Right. But the proper solution is bi-temporal modeling. Two time dimensions: **valid time** (when the meter reading was true in the real world) and **transaction time** (when we recorded it in our system). With both, you can answer "what did we know, as of when?"

**Dr. Marcus Thompson**: That second dimension is important for the late-arrival problem. A meter in rural Norway sends a delayed reading — it has a valid time of yesterday but arrives today. Right now, we write it with today's timestamp. If Ofgem asks what we knew at 14:32:07 yesterday, we'd show it, even though we didn't know it yet at that time.

**[you]**: Exactly. That's an evidentiary problem. Regulatory data requires provable knowledge-at-time, not just fact-at-time.

**Dr. Marcus Thompson**: So the architecture needs both.

**[you]**: Yes. And Apache Iceberg's snapshot model gives us transaction time for free — every snapshot is an immutable record of what the table looked like at a point in time. We need to add valid time as a first-class column.

### Problem Statement

Crestline Energy cannot answer a regulatory request for point-in-time grid state because its current system stores only current state and event logs that are unindexed at the required timestamp precision. A bi-temporal data model is needed that records both when a measurement was true (valid time) and when it was recorded in the system (transaction time), enabling precise regulatory queries.

---

### Explicit Requirements

1. Answer Ofgem's specific query (grid state at 14:32:07 UTC, March 3rd) within 3 days using existing archive data
2. Design a bi-temporal data model that makes such queries instant (sub-minute) going forward
3. Separate valid time from transaction time: a late-arriving meter reading must record both the measurement time and the reception time
4. Iceberg snapshot retention: regulatory data must be queryable at any point in the last 7 years (NERC CIP retention requirement)
5. Query API: regulators need a self-service interface for point-in-time queries (subject to authentication)
6. The design must survive schema evolution: new meter types added in the future must also support temporal queries retroactively

---

### Hidden Requirements

- **Hint**: Dr. Thompson identified the evidentiary problem: late-arriving readings mixed into historical state create a provability gap. If Ofgem challenges whether Crestline "knew" about an anomaly at 14:32:07, you need to prove what was in your system at exactly that timestamp — not what the meters were reading. This is the transaction time dimension. Your design must preserve it.

- **Hint**: "7 years of queryable history" for 50M meters × 96 readings/day is a massive dataset. Iceberg's time-travel can answer queries against historical snapshots — but keeping 7 years of fine-grained snapshots has a storage cost. What is the right snapshot retention policy (daily snapshots? weekly? event-triggered?) to balance query precision with storage cost?

### Constraints

- Historical archive: 3 years of raw event files in S3, unindexed, 18TB
- Ofgem deadline: 30 days from letter receipt
- Regulatory retention requirement: 7 years of queryable state history (NERC CIP / Electricity Act)
- Immediate query requirement: future regulatory queries must complete in < 5 minutes
- Late-arriving data: up to 4% of readings arrive more than 1 hour late
- Snapshot storage budget: increase allowed (Iceberg historical snapshots), but must be costed
